## Overview

SolidFI defines a small set of composable primitives for expressing data transformations. The central insight is that data is simple; the interesting complexity lives in the transitions between data. SolidFI makes those transitions first-class: named, typed, and composable.

Determinism is a first-class value. Composition is the primary mechanism. Error handling is opt-in.

Level 1 has four core primitives: `Converter`, `Chain`, `Transform`, and `Pipeline`. These four are the heart of SolidFI â€” nearly everything that follows is either an abstraction or generalization of them.

### Principle: Names for Important Things

SolidFI uses named marker types and named primitives so that every important concept has an identity. Anonymous types, functions, and compositions can make readability, debuggability, and reasoning challenging.

A name is not just a label â€” it is a place to attach relationships, invariants, and meaning. `Parameters` exists as a named marker so that the concept of "contextual data flowing into a conversion" has a home in the spec, even before its internal structure is defined.

This principle applies throughout: if something matters, it has a name.

## Primitive Summary

**Structural Concepts**

- **`Transform<T>`** â€” Takes a `T` and produces a `T`. Cannot fail; degrades to identity if change is not possible. No parameters.
- **`Pipeline<T>`** â€” An ordered composition of `Transform<T>` instances. Is itself a `Transform<T>`. Rejected transforms pass through unchanged.
- **`Converter<T,U,P>`** â€” Takes a `T` and produces a `U`. May fail. Accepts optional user-defined parameters `P` for routing and dispatch.
- **`Chain<T,U,P>`** â€” An ordered composition of `Converter` instances. Is itself a `Converter<T,U,P>`. Short-circuits on the first accepting converter.
 
**State Concepts**

- **`Delta<T>`** â€” Represents the change between two states of `T`. A named, first-class concept for the transition itself.
- **`Differencer<T>`** â€” `Converter<T, Delta<T>, T>` â€” Given a current state and a previous state (as `P`), produces a `Delta<T>`.
- **`Applicator<T>`** â€” `Converter<T, T, Delta<T>>` â€” Given a current state and a `Delta<T>` (as `P`), produces the next state.

**Runtime Concepts**

- **`Graph`** â€” An unordered, potentially cyclic registry of `Converter<*,*>` instances. Answers the static query `hasPath<T,U>() -> bool`: does a path exist between these types?
- **`Traversal<Graph,T,U,P>`** - Given a `Graph`, find some path from T to U.
- **`Solver<Graph,T,U,P>`** â€” Is itself a `Converter<T,U,P>`. Given a `Graph`, finds a specific `U` for an instance of `T`, with parameters `P`.

**Peripheral L1 Concepts**

- **`Predicate<T>`** â€” Defines the `accepts`/`rejects` filtering contract. Both methods are optional; defaults accept everything and reject nothing.
- **`Parameters`** â€” Marker type for user-defined contextual data passed into `Converter` via `fetch`. The framework never inspects it.
- **`Literal<T>`** â€” Captures a `T` and produces it regardless of input. Satisfies both `Transform<T>` and `Converter<any,T>` simultaneously. Also known as `Closed<T>`.

### Transform\<T\>

*Satisfies `Predicate<T>`.*

A `Transform` takes a value of type `T` and produces a value of type `T`. Input and output are the same type. A `Transform` cannot fail â€” it MUST return the original `T` if change is not possible. This is the degradation floor.

Transforms do not take parameters. If contextual parameters are needed, use `Converter<T, T, P>` instead â€” the parameter is a signal that the concern is conversion, not transformation.

**Operations:**
- `accepts(T) -> boolean` â€” default: `true`
- `rejects(T) -> boolean` â€” default: `false`
- `apply(T) -> T` â€” perform the transformation. MUST return a T.

**Invariants:**
- `accepts` and `rejects` follow the `Predicate<T>` rules.
- `apply` MUST return a value of type T.
- A transform that does not accept a T MUST return the original value unchanged.

---

### Pipeline\<T\>

*Implements: `Transform<T>`. Satisfies `Predicate<T>`.*

A `Pipeline` is a concrete, ordered composition of `Transform<T>` instances. It is itself a `Transform<T>` â€” the Composite rule. To any caller holding a `Transform<T>` reference, a `Pipeline` is indistinguishable from a single transform.

Pipeline takes no parameters â€” if parameters are needed, use `Chain<T, T, P>` instead.

A rejected transform acts as a passthrough â€” the input is forwarded unchanged to the next transform in priority order. Rejection is not failure; it is abstention.

**Operations:**

Inherits from `Transform<T>`:
- `accepts(T) -> boolean`
- `rejects(T) -> boolean`
- `apply(T) -> T` â€” runs the pipeline. Alias: `run(T)`

Own operations:
- `run(T) -> T` â€” friendly alias for `apply`. Prefer `run` when calling a known Pipeline directly.
- `install(priority: int, name: string, Transform<T>) -> void`
- `remove(name: string) -> void`
- `replace(name: string, Transform<T>) -> void`

**Invariants:**
- A `Pipeline` MUST degrade to identity if no installed transform accepts the input.
- A rejected transform acts as a passthrough â€” the input is forwarded unchanged to the next transform in priority order.
- Execution, priority, naming, and grouping behavior follows the L0 `Composite` substrate. See `Chain` invariants and L0 Composite.

### Converter\<T, U, P\>

*Satisfies `Predicate<T>`.*

A `Converter` takes a value of type `T` and produces a value of type `U`. `T` and `U` are usually different types â€” this is a genuine conversion, not a transformation. `P` is an optional user-defined parameter type, defaulting to empty/nothing. Users define their own `P` to carry contextual data for routing and dispatch. The framework never inspects `P` â€” it only passes it through to `fetch`.

Because `T` and `U` are distinct, failure is possible â€” there is no identity fallback when types differ. How failure is represented is TBD â€” candidates include encoding failure in the return type (`U|null`, `optional<U>`) or other mechanisms.

**Operations:**
- `accepts(T) -> boolean`
- `rejects(T) -> boolean`
- `fetch(T, P) -> U` â€” perform the conversion. Async. May fail.

**Invariants:**
- `accepts` and `rejects` follow the `Predicate<T>` rules.
- `fetch` MAY be asynchronous.
- `fetch` MAY fail. How failure is represented is TBD.

### Chain\<T, U, P\>

*Implements: `Converter<T, U, P>`. Satisfies `Predicate<T>`.*

A `Chain` is a concrete, ordered composition of `Converter` instances. It is itself a `Converter<T, U, P>` â€” the Composite rule. To any caller holding a `Converter<T,U,P>` reference, a `Chain` is indistinguishable from a single converter. It can be swapped in or out at runtime transparently.

A `Chain` is strictly typed at its endpoints â€” it accepts `T` and produces `U`. Intermediate types within the chain are an implementation detail invisible to callers. At these endpoints, the chain may optionally have an input `Transform<T>` and output `Transform<U>` applied before and after conversion respectively. This allows for arbitrary transforms to be explictly attached to certain conversions, without requiring `fetch` to know about them explicitly. Transforms can be added via self-definition, or by others added at runtime.

It is recommended that any factory code that creates a `Chain` returns it as a `Converter` â€” callers should never need to know a `Chain` is involved. Chains are intended to be assembled at startup and reused, however it is entirely valid to create and dispose of a `Chain` per use if needed.

**Operations:**

Inherits from `Converter<T, U, P>`:
- `accepts(T) -> boolean`
- `rejects(T) -> boolean`
- `fetch(T, P) -> U` â€” execute the chain. Alias: `resolve(T, P)`

Own operations:
- `resolve(T, P) -> U` â€” friendly alias for `fetch`. Prefer `resolve` when calling a known Chain directly.
- `install(priority: int, name: string, Converter<T, U>) -> void`
- `remove(name: string) -> void`
- `replace(name: string, Converter<T, U>) -> void` â€” equivalent to `remove` followed by `install`, keeping priority unchanged.

Own members:
- `prepare` - an optional `Transform<T, P>` that runs before dispatching input to fetch, can be a `Pipeline<T, P>`
- `finalize` - an optional `Transform<U, P>` that runs after receiving a successful result from fetch, can be a `Pipeline<U, P>`

**Invariants:**
- Execution order is determined by `priority`. Priority SHOULD be unique for determinism â€” duplicate priorities result in undefined ordering within that priority level.
- Names are group keys, not identity keys. Multiple entries MAY share a name and are treated as a group â€” `remove(name)` removes all entries with that name.
- A name MAY be abstracted into a `Strategy` type in a future revision.
- The first converter that `accepts` and does not `reject` is attempted via `fetch`. If `fetch` succeeds, the chain short-circuits and returns the result. If `fetch` fails, the chain continues to the next converter in priority order.
- If no converter succeeds, the chain fails.
- `accepts` and `rejects` are an optimization â€” they avoid calling `fetch` when the outcome is already known. A converter with no `accepts` or `rejects` implemented will always be attempted via `fetch`.
- `install` accepts any `Converter<T,U>` â€” including `Inverter`, `Goto`, or any other L1 subtype, or another `Chain`.

*Priority, naming, and grouping behavior is defined by the L0 `Composite` substrate and shared with `Pipeline`. See L0 Composite.*

The filtering rule for both `Chain` and `Pipeline`:

```
if (!rejects(T) && accepts(T)) â†’ attempt fetch
```

`rejects` is evaluated first. A converter that passes the filter is *attempted* â€” not selected. Selection is only final when `fetch` succeeds. If `fetch` fails, the chain continues to the next converter in priority order.

This allows orthogonal concerns to compose without coupling â€” one piece declares what it refuses, another declares what it handles, neither needs to know about the other. The chain tries each candidate in order until one succeeds.

### Graph

*No type parameters. Holds an unordered, heterogeneous, potentially cyclic set of `Converter<*,*>` instances.*

A `Graph` is a registry of `Converter` instances across arbitrary type pairs. It represents the full set of conversions available to a system and answers one structural question: given a source type `T` and a target type `U`, does a path exist through the installed converters?

A `Graph` does not execute conversions. Execution is `Solver`'s concern. `Graph` answers "is this possible?"; `Solver` answers "make it happen."

**Operations:**
- `hasPath<T,U>() -> bool` â€” returns `true` if a path from `T` to `U` exists through the installed converters.
- `install(Converter<*,*>) -> void`
- `remove(Converter<*,*>) -> void`

**Invariants:**
- `hasPath` is a static, structural query â€” it MUST NOT execute any converter and MUST NOT consider `P`.
- `hasPath` MAY return `true` even if no specific instance of `T` can be converted â€” reachability is a graph property, not a guarantee of execution success.
- The graph MAY contain cycles. Cycle detection and avoidance is the responsibility of traversal â€” i.e., `Solver`.
- Installed converters are unordered. No priority applies at the `Graph` level.
- A `Graph` SHOULD be assembled at startup and treated as stable. It is not designed for high-frequency mutation.

### Solver\<T, U, P\>

*Implements: `Converter<T, U, P>`. Satisfies `Predicate<T>`.*

A `Solver` is a `Converter<T, U, P>` backed by a `Graph`. Given a value of type `T` and parameters `P`, it uses the `Graph` to find a path to `U` and executes it.

Because `Solver` implements `Converter<T, U, P>`, it composes transparently wherever a `Converter` is expected â€” including inside a `Chain`. To any caller holding a `Converter<T,U,P>` reference, a `Solver` is indistinguishable from a single converter.

**Operations:**

Inherits from `Converter<T, U, P>`:
- `accepts(T) -> boolean`
- `rejects(T) -> boolean`
- `fetch(T, P) -> U` â€” finds a path through the `Graph` and executes it.

Own operations:
- `constructor(Graph)` â€” binds this `Solver` to a specific `Graph`.

**Invariants:**
- `P` flows into individual converters along the path, consistent with `Converter` semantics. The `Graph` itself is not inspected with `P`.
- `Solver` is responsible for cycle avoidance during traversal. The primary mechanism is `Chain`-based dispatch: where multiple converters exist for a given type, a `Chain` selects among them via `P`, routing the traversal without re-entering the same path.
- If no path exists, `fetch` MUST fail.
- If a path exists but execution fails, `fetch` MUST fail.
- A `Solver` MUST NOT modify the `Graph` it holds.

## Peripheral L1 Concepts

### Predicate\<T\>

*L1 concept. May be satisfied formally via interface or informally by providing the methods inline.*

A `Predicate` defines the filtering contract for any SolidFI primitive that handles inputs. Both methods are optional â€” the defaults constitute a trivially true predicate that accepts everything and rejects nothing.

**Operations:**
- `accepts(T) -> boolean` â€” default: `true`. Does this handler claim this input?
- `rejects(T) -> boolean` â€” default: `false`. Does this handler explicitly refuse this input?

**Invariants:**
- MUST be stateless and synchronous.
- MUST be callable more than once without side effects.
- SHOULD be pure and dependent only on T. If the decision requires inspection of any other data, that logic belongs in `fetch`, signaled via a null or failure return value.
- MUST NOT depend on P.
- Callers are NOT required to call `accepts` or `rejects`. They are a fast-path optimization to avoid extra processing â€” not a mandatory gate.
- `rejects` is an optimization gate â€” a handler that rejects is skipped entirely and never tried.

`Predicate<T>` may be satisfied formally via interface or informally by providing the methods inline. Any type may satisfy it, not just SolidFI primitives.

---

### Parameters

*Marker type. No required internal structure.*

`Parameters` is a user-defined type that carries contextual data into a `Converter` via `fetch(T, P)`. The framework never inspects or constrains `Parameters` â€” it only passes it through.

In TypeScript the natural default is `{}` â€” an empty object, compatible with anything. Other languages may use `void`, a null type, or a missing type argument.

`Parameters` has relationships to other concepts that will be defined as the spec matures. It is named here to establish it as a first-class concept rather than an anonymous type variable.

---

### Literal\<T\>

*Implements: `Transform<T>` and `Converter<any, T>`. Satisfies `Predicate<T>`.*

A `Literal<T>` captures a value of type `T` and produces it regardless of input. It simultaneously satisfies both `Transform<T>` and `Converter<any, T>`, making it injectable anywhere either is expected â€” without the caller needing to choose.

**Common uses:**
- Inject a constant or variable into a `Chain` or `Pipeline` without changing its type contract.
- Adapt an existing value into a composition that expects a transform or converter.
- Stand-in during development before a real implementation exists.

**Operations:**
- `constructor(T)` â€” captures the initial value.
- `get() -> T` â€” retrieve the current value.
- `set(T) -> void` â€” update the captured value.
- `fetch(T, P) -> T` â€” delegates to `get`. Input is ignored.
- `apply(T) -> T` â€” delegates to `get`. Input is ignored.

**Invariants:**
- MUST always produce a T.
- `fetch` and `apply` MUST NOT depend on the input value.
- `accepts` returns `true` by default, `rejects` returns `false`.
